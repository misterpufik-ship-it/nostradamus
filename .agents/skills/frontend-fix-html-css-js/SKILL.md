# frontend-fix-html-css-js

Используй этот skill, когда нужно проверить и аккуратно исправить проблемы фронтенда: HTML-разметку, CSS, JavaScript, адаптивность, мобильную версию, визуальные баги, кликабельность, переполнения, наложения блоков и ошибки в браузере.

Главный принцип: сначала понять текущий сайт и его ограничения, затем вносить минимальные точечные правки. Не переписывать проект целиком и не ломать CMS.

---

## 1. Перед началом

Сначала изучи проект:

1. Прочитай `AGENTS.md`, если он есть.
2. Найди фактический корень сайта и основную рабочую папку.
3. Определи, какие файлы отвечают за страницу: PHP/HTML-шаблоны, CSS, JS, подключаемые фрагменты, CMS-теги.
4. Найди способ локального предпросмотра: PHP-сервер, статичный preview, готовый HTML-снимок или dev-server.
5. Не начинай массовые правки, пока не понял структуру.

Для этого проекта Garden Lounge учитывать:

```text
Основной корень Codex:

Основная рабочая папка:

```

## 2. Что проверять

Проверяй сайт как пользователь, а не только как код:

- первый экран и шапку;
- меню и якорные ссылки;
- hero-блок;
- карточки, галереи, слайдеры, формы;
- кнопки, ссылки, hover/focus-состояния;
- мобильную навигацию;
- переполнение текста;
- горизонтальный скролл;
- наложение блоков;
- отступы между секциями;
- читаемость текста на фоне;
- загрузку изображений, видео, шрифтов и иконок;
- ошибки JavaScript в консоли;
- некорректные пути к файлам;
- поведение на ширинах 320, 375, 430, 768, 1024, 1366 и выше.

---

## 3. Как исправлять

Правки должны быть аккуратными:

1. Сначала найди существующее место в CSS/JS/шаблоне.
2. Добавляй правку рядом с уже существующей логикой.
3. Не дублируй большие блоки стилей без необходимости.
4. Не добавляй новые библиотеки, если проблему можно решить текущими средствами.
5. Не меняй HTML-структуру радикально, если достаточно CSS-правки.
6. Не удаляй CMS-теги и серверные вставки.
7. Не трогай файлы `couch/`, `node_modules/`, кеши, загрузки и конфиги без явной задачи.
8. Не запускай автоформатирование всего проекта, если задача точечная.

Если в проекте есть отдельный CSS для полировки или локальных улучшений, предпочитай его для небольших визуальных исправлений. Для Garden Lounge это:

```text
public_html/admiralteyskaya/site-polish.css
```

Если проблема находится в базовой системе стилей и уже описана в `main.css`, можно править `main.css`, но только точечно.

## 4. Особые правила для PHP/CouchCMS

В PHP-файлах могут быть теги CouchCMS:

```html
<cms:template>
<cms:editable>
<cms:pages>
<cms:embed>
<cms:show>
```

С ними обращаться осторожно:

- не удалять;
- не переименовывать поля без необходимости;
- не менять `masterpage`, `name`, `group`, `repeatable`, если это не часть задачи;
- не ломать `require_once('couch/cms.php')`;
- не удалять `COUCH::invoke()`;
- помнить, что часть контента может приходить из базы CouchCMS, а не из файла.

После правок обязательно проверить визуально, если есть доступный способ.

## 6. Минимальная техническая проверка

Если менялись PHP-файлы и доступен PHP, проверить синтаксис измененных файлов:

```powershell
php -l path\to\file.php
```

Если менялись CSS/JS/HTML/JSON/MD в папке `admiralteyskaya`, можно выполнить:

```powershell
npm run format:check
```

Не запускать `npm run format` без отдельной необходимости, чтобы не переформатировать старые файлы и не создать лишний шум.

## 7. Отчет после работы

В финальном ответе кратко указать:

1. Что найдено.
2. Что исправлено.
3. Какие файлы изменены.
4. Как проверено.
5. Что осталось проверить вручную, если полноценный PHP/CMS-запуск недоступен.

Ответ должен быть коротким и понятным для владельца сайта, без лишней технической воды.
